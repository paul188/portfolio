## Overview

BIDS-Eye is a natural-language search engine over **OpenNeuro**, the primary open-access repository for human neuroimaging data (MRI, fMRI, EEG, MEG, PET). It answers queries like *"datasets with ≥ 50 left-handed participants who did an n-back task with fMRI"* by crawling ~1,000 datasets from AWS S3, indexing them into a unified PostgreSQL database, and translating free-text queries into SQL through a three-tier RAG pipeline backed by a 1,466-concept biomedical ontology.

---

## 1. OpenNeuro and the BIDS Standard

### OpenNeuro

OpenNeuro (`openneuro.org`) hosts thousands of neuroimaging datasets from research groups worldwide. Datasets are identified by accession numbers (e.g. `ds000228`, `ds004504`). Raw data lives on AWS S3 under the `openneuro.org` bucket, publicly accessible via anonymous boto3 for most datasets.

### The BIDS Standard

The Brain Imaging Data Structure (BIDS) is a community standard defining how a neuroimaging dataset must be organised on disk. Its core insight: **file paths and JSON sidecars *are* the metadata** — no separate database needed.

```
ds000228/
├── dataset_description.json      ← name, authors, license, BIDS version
├── participants.tsv               ← one row per subject: age, sex, handedness, diagnosis
├── sub-001/
│   ├── anat/
│   │   └── sub-001_T1w.nii.gz
│   └── func/
│       ├── sub-001_task-rest_bold.nii.gz
│       └── sub-001_task-rest_bold.json   ← sidecar: TR, slice timing, etc.
└── derivatives/
    └── fmriprep/sub-001/...
```

Every property is encoded as **entities** in the filename: `sub-` (subject), `ses-` (session), `task-` (task name), `run-` (run number), `acq-` (acquisition protocol). The **suffix** (`bold`, `T1w`, `eeg`) encodes what kind of data it is; the **datatype folder** (`func/`, `anat/`, `dwi/`, `fmap/`) encodes the modality category.

This rigid structure makes BIDS-Eye possible: because every dataset follows the same conventions, a unified SQL database can be built across thousands of datasets without bespoke parsing.

## 2. The BIDS-SQL Database Schema

### Design Principles

pybids (the reference Python BIDS library) builds an in-process SQLite index of one dataset at a time. BIDS-Eye generalises this to a **persistent PostgreSQL database spanning thousands of datasets simultaneously**.

Three design principles govern the schema:

1. **Hot entities get real columns** — properties most commonly filtered (`subject`, `session`, `task`, `suffix`, `datatype`, `sex`, `handedness`, `diagnosis`) are first-class indexed columns.
2. **Cold entities go into JSONB overflow** — BIDS has ~30 secondary entities (`acq`, `ce`, `dir`, `echo`, `hemi`, `space`, etc.) and arbitrary sidecar metadata. These land in `other_entities` (JSONB with GIN index), queryable via `->` operators without schema migrations.
3. **Upserts are idempotent** — every write uses `ON CONFLICT … DO UPDATE`, so re-crawling a dataset refreshes rows rather than duplicating them.

## 3. The Crawlers

### Why Crawling is Non-Trivial

OpenNeuro hosts ~1,000 datasets. Cloning all raw imaging files (`.nii.gz`) would require terabytes. BIDS-Eye uses **ghost indexing**: download only metadata files (`.json`, `.tsv`), create empty stub files for imaging data, run the full pybids pipeline on the local mirror, mark all objects `is_remote=True`.

### Primary Path — S3 Anonymous Access

**Step 1 — S3 listing.** Anonymous boto3 client paginates through the `openneuro.org` bucket. No credentials required for public datasets.

**Step 2 — Mirror building** (cached at `~/.bids-sql/openneuro/&#123;accession_id&#125;/`)
- `.json` and `.tsv` files: downloaded with full content; JSON files are repaired for common authoring mistakes (trailing commas, control characters, encoding issues)
- `.nii.gz`, `.mgz`, other imaging files: created as **empty stubs** — 0 bytes is enough for pybids to see them in the tree
- `._*` AppleDouble files (macOS artifacts): skipped entirely
- Smart caching: existing files are reused unless they are 0-byte or JSON-corrupt

**Step 3 — pybids indexing.** `BIDSLayout` is run on the local mirror, parsing every filename, validating entities, and building an internal index. BIDS-Eye then upserts that index into PostgreSQL.

**Step 4 — Post-indexing stamps.** `source_type="openneuro"`, `accession_id`, `remote_url=s3://openneuro.org/&#123;accession_id&#125;` are set. Promoted metadata fields (`authors`, `license`, `doi`, `funding`, `description_text`) are extracted from `dataset_description.json`.

### Fallback — GraphQL API

Some datasets are access-restricted on S3. The fallback hits `https://openneuro.org/crn/graphql` and synthesises synthetic participant and object rows. `source_type="openneuro_graphql"` distinguishes synthetic from real records in queries.

### Scheduler

APScheduler runs `run_crawl_cycle()` on a configurable interval (default: 1 hour). One dataset is processed at a time via `asyncio.Semaphore(1)`. Crawler state is exposed at `GET /api/crawler/status`.

## 4. Data Normalisation and value_mappings.yaml

### The Problem

Raw BIDS datasets are terminologically inconsistent. The same diagnosis appears as `"MDD"`, `"major depressive disorder"`, `"clinical depression"`, `"depression"`, and `"296.20"` across different datasets. Without normalisation, no cross-dataset query is possible.

`RAG/value_mappings.yaml` is the **unified ontology** mapping all surface forms onto canonical codes — **1,466 concepts across 8 categories** (diagnosis, task, suffix, datatype, sex, handedness, sidecar_fields, participant_extra_fields).

### Structure: Flat SKOS

```yaml
diagnosis:
  major_depressive_disorder:
    label: "major depressive disorder"
    standard_code: "major_depressive_disorder"
    broader: [mood_disorders]
    synonyms:
      - term: "MDD"
        weight: 0.85
      - term: "296.20"        # ICD-9 code as synonym
        weight: 0.6
    codes: [mdd, depression, depressive_disorder]
```

Flat (non-nested) layout enables O(1) lookup and **polyhierarchy** — a concept can have multiple parents via `broader: [parent_a, parent_b]`.

The retriever raises the fuzzy-match threshold for low-confidence synonyms:

```python
effective_threshold = 80 + (1.0 - weight) × 8
```

A synonym with weight 0.5 requires fuzzy score 84 (not 80) to resolve.

### Synonym Generation (expand_synonyms.py)

Synonyms are mined from three corpus sources and validated by LLM: PostgreSQL `description_text`, CrossRef REST API (titles and abstracts via DOI), and OpenAlex REST API (neuroscience subset, 250M+ works). LLM backend: Claude Sonnet (preferred) → Gemini 2.5 Flash (fallback).

Weight: `0.6 × (fraction_of_texts_containing_term) + 0.4 × log_scaled_frequency`, floored at 0.1.

### Collision Detection

A collision is a synonym in concept A that is also the label or high-weight (≥ 0.7) synonym of concept B. Detection uses RapidFuzz WRatio. The `task` field alone had **239 CRITICAL** and **1,075 HIGH** collisions — most commonly a parent concept mistakenly claiming a child's label. `fix_collisions.py` applies 299 curated removals as line-level text surgery.

## 5. End-to-End NLQ-to-SQL Pipeline

### Why Not SQLCoder or Biomedical Language Models?

SQLCoder (fine-tuned Code Llama) was evaluated but abandoned: requires a GPU inference endpoint, has no neuroscience vocabulary, and with the scaffolding needed to give it vocabulary, Gemini performs equivalently without a GPU. Biomedical generative models (BioBERT, PubMedBERT) are not instruction-tuned and would require substantial fine-tuning.

BioLORD-2023-C and SapBERT are **retained for RAG retrieval** where they excel at entity linking — but only for retrieval, not generation.

**Current architecture:** Gemini for high-level reasoning + domain-specific RAG (`value_mappings.yaml` + biomedical embeddings) for vocabulary resolution.

### Example Query

> *"datasets with resting-state fMRI and at least 20 autistic participants"*

### Stage 1 — Entity Extraction

`RAG/llamaindex_extraction.py` — Gemini 2.5 Flash returns a structured `UserQueryEntities` Pydantic object:

```python
UserQueryEntities(
    diagnosis=["autism"],
    task=["resting state"],
    datatype=["fMRI"],
    suffix=[], sex=[], handedness=[], ...
)
```

### Stage 2 — RAG Resolution: Surface Forms → Canonical Codes

**This is not standard RAG.** Most RAG systems retrieve text chunks for an LLM to paraphrase. This one resolves surface forms to canonical database keys — the string `autism_spectrum_disorder` that goes directly into a SQL `WHERE` clause. The LLM never touches the ontology vocabulary; it only sees pre-resolved codes. This is closer to **biomedical entity linking** than to generative RAG.

`RAG/retriever.py` runs a three-tier cascade:

```
User term
    │
    ▼
Tier 1 — Exact match       O(1) dict lookup against leaf_db / group_db
    │ miss
    ▼
Tier 2 — Fuzzy (WRatio)    weight-adjusted threshold: 80 + (1.0 − w) × 8
    │ score < threshold
    ▼
Tier 3a — BioLORD-2023-C   cosine ≥ 0.75  (SNOMED-CT / MedDRA)
    │ miss
    ▼
Tier 3b — SapBERT           cosine ≥ 0.65  (UMLS 4M+ entities)
    │ miss
    ▼
VOCAB MISS → ILIKE hint injected into SQL
```

**Tier 1 — Lookup structures.** `yaml_to_llamaindex.py` builds six in-memory dicts on startup via a two-pass parse. The first pass registers all leaf concepts; the second does a DFS over the `broader` DAG to populate `group_db` — so a group-node lookup returns all descendant canonical codes, not just the group itself.

**Tier 2 — Why WRatio, and the weight-adjusted threshold.** WRatio is a meta-scorer that tries plain Levenshtein, token-sorted, and token-set variants and returns the highest — the right choice when terms vary in both word order (`"fMRI resting state"` vs `"resting state fMRI"`) and abbreviation length (`"ADHD"` vs `"attention deficit hyperactivity disorder"`). The threshold is not fixed at 80: it rises as synonym weight falls.

```python
effective_threshold = 80 + (1.0 - synonym_weight) × 8
```

`"autism"` (weight 1.0) fires at score ≥ 80. `"developmental disorder"` (weight 0.3) requires score ≥ 85.6 — a low-confidence synonym only matches if the user typed it nearly verbatim. This encodes *how exclusively a term belongs to a concept*.

**Tier 3 — Why BioLORD before SapBERT.** BioLORD is trained on curated SNOMED-CT / MedDRA concept pairs — very high precision for clinical terms, but narrower coverage. SapBERT covers ~4M UMLS entities with higher recall but lower precision. Running BioLORD first with the higher threshold (0.75) captures precise clinical matches; SapBERT at 0.65 catches the longer tail. Both models have embeddings pre-computed for all ~29 000 synonym surface forms and stored as `.npz` files keyed on a hash of `value_mappings.yaml` — only the user's term is embedded live at query time.

**Multi-field simultaneous search.** `"fMRI"` is simultaneously a `datatype` (`func/` folder) and a `suffix` (`bold`). The resolver searches `datatype`, `suffix`, and `task` in parallel and returns all hits. Pure modality abbreviations are explicitly filtered from the `task` field to prevent `"fMRI"` → a task concept that has `"fMRI"` as an incidental synonym.

**Hierarchy pruning.** After resolution, if a result set contains both a concept and one of its ancestors, the ancestor is dropped — a DFS up each code's `broader` chain removes any parent also present in the set. Without this, `WHERE o.datatype IN ('mri', 'func', 'functional_mri')` would match far more data than the user intended.

**Vocabulary miss handling.** When all tiers fail, the preprocessor pre-computes an `ILIKE` hint during Stage 1 and holds it in the `QueryPlan`. If the LLM subsequently generates a `WHERE` filter using a made-up code, that filter is stripped and the `ILIKE` is injected instead — vocab misses degrade to approximate text search rather than SQL errors or empty results.

**Resolved output:**
```json
{
  "diagnosis": ["autism_spectrum_disorder"],
  "task":      ["resting_state"],
  "datatype":  ["func"],
  "suffix":    ["bold"]
}
```

### Stage 3 — SQL Generation

`backend/services/text_to_sql.py` — resolved codes, full schema DDL, and few-shot examples assembled into a Gemini prompt. `RAG/join_paths.yaml` provides routing rules for which codes map to which table/column and how multi-value filters combine.

**Generated SQL:**
```sql
SELECT DISTINCT d.id, d.name, d.accession_id,
       COUNT(DISTINCT p.participant_id) AS n_participants
FROM bids_datasets d
JOIN bids_participants p ON p.dataset_id = d.id
JOIN bids_objects o ON o.dataset_id = d.id
WHERE p.diagnosis = 'autism_spectrum_disorder'
  AND o.task = 'resting_state'
  AND (o.datatype = 'func' OR o.suffix = 'bold')
GROUP BY d.id, d.name, d.accession_id
HAVING COUNT(DISTINCT p.participant_id) >= 20
ORDER BY n_participants DESC;
```

### Stage 4 — Execution and Error Correction

`backend/routers/query.py` — on SQL failure:
1. **Session rollback** immediately (critical: PostgreSQL enters an error state without this)
2. **Gemini correction** — receives the original question, failed SQL, full schema DDL, and the PostgreSQL error message
3. **Retry** — corrected SQL is executed; response is flagged `self_corrected: True`
4. **No infinite loop** — if the correction also fails, the original error is returned

All LLM calls use exponential backoff across four model variants: `gemini-2.5-pro` → `gemini-2.5-flash` → `gemini-flash-lite` → `gemini-deep-think`.

**Regression testing:** 77 hand-crafted queries tested after any pipeline change, checking minimum result count, required SQL fragments, and forbidden patterns (e.g. ILIKE on a canonical-code column).
