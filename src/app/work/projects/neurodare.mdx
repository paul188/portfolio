## Overview

NeuroDaRe (GesundheitsDAtentreuhand-REallabor) is a funded research prototype for a **Healthcare Data Trust** — an institutional intermediary that allows medical data providers and researchers to exchange sensitive radiological data without either party having unmediated access to the other's systems. Data providers upload DICOM studies; data users request access; the Data Trust validates, pseudonymises, and brokers the exchange under ethical and legal constraints.

The platform is built around three interlocking concerns: **privacy** (DICOM pseudonymisation with an isolated PHI database), **security** (multi-layer runtime threat detection, antivirus scanning, VPN-gated administration), and **auditability** (centralised ELK logging, full Alembic migration history, Keycloak event webhooks). It runs as 17 Docker containers with a single `make up-build` entrypoint and is deployed at [neurodare.org](https://www.neurodare.org).

## 1. The Data Trust Model

NeuroDaRe implements a three-role trust model. Keycloak enforces role-gating at the OIDC token level — every API request carries a JWT whose `realm_access.roles` claim is verified by FastAPI middleware before any route handler executes.

The platform runs **17 Docker containers** on a shared `neuro_network` bridge. All external traffic enters through nginx, which terminates TLS and reverse-proxies to the appropriate upstream. No application container is directly exposed to the internet.

              ├─ /auth/*   → Keycloak

## 3. DICOM Upload Pipeline

Uploading a radiological study triggers a six-step background pipeline (`app/services/upload_session/background_tasks.py`). Each step must pass before the next executes; any failure aborts the session and notifies the provider.

<ClickableImage src="/images/projects/neurodare-dicom-pipeline.png" alt="NeuroDaRe DICOM upload pipeline — validation, virus scan, pseudonymisation, Orthanc ingestion" />

1. **File arrives** — `POST /api/provider/uploads/` receives the multipart upload and writes files to a temporary session directory.
2. **DICOM validation** — pydicom-validator checks conformance. Non-DICOM files are rejected entirely; the platform accepts only DICOM.
3. **Virus scan** — every file is passed to the ClamAV daemon via clamd socket. Infected files abort the session.
4. **Pseudonymisation** — idiscore replaces all PHI-bearing DICOM tags (`PatientName`, `PatientID`, `BirthDate`, etc.) with `ANON_XXXXXXXX` values. The original-to-pseudonym mapping is written to the **PHI database** — the only location in the entire system where original patient identifiers are retained (see §5).
5. **Orthanc ingestion** — pseudonymised files are pushed to Orthanc via DICOMweb HTTP. Orthanc writes the pixel data to the S3 object store and indexes the study in its own PostgreSQL instance.
6. **Metadata persistence** — a dataset record (study metadata, pseudonymous identifiers, access policy) is written to the metadata PostgreSQL database, making the study discoverable to data users.

## 4. Authentication Flow

NeuroDaRe uses Keycloak 26 as a full OIDC provider. FastAPI acts as a confidential OAuth2 client — it never handles passwords directly.

<ClickableImage src="/images/projects/neurodare-auth-flow.png" alt="NeuroDaRe OIDC authentication flow — Keycloak code exchange via FastAPI" />

1. `GET /api/login` — FastAPI constructs the Keycloak authorisation URL (including PKCE parameters) and redirects the browser.
2. **Keycloak authentication** — the user authenticates at `/auth/realms/fastapi_realm/`. If email 2FA is enrolled (via the `keycloak-2fa-email-authenticator` plugin), a one-time code is sent and verified here.
3. `GET /api/callback?code=...` — Keycloak redirects back to FastAPI with an authorisation code.
4. **Token exchange** — FastAPI exchanges the code for an access token and ID token via the Keycloak token endpoint, then stores the session in Redis.
5. **Role dispatch** — FastAPI reads the `realm_access.roles` claim and redirects to the role-appropriate dashboard (`/provider`, `/user`, or a pending-approval screen).

Session cookies are `HttpOnly`, `Secure`, and `SameSite=Lax`. CSRF tokens are signed separately with a distinct secret key.

The PHI database is the architectural linchpin of the privacy model. A data breach of `metadata_db` or `orthanc_db` exposes only pseudonymous identifiers — re-identification requires a separate attack on `postgres_phi`, which carries its own credentials, its own Docker volume, and is never queried by any route other than the pseudonymisation step of the upload pipeline.
